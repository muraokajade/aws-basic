# リベンリOCRデモページ作成_進捗報告_2026_7_10

## 1. 7月9日の実施内容

7月9日は「AWS Lightsail超入門」として、AWS Lightsailの概要と、Dockerアプリを動かす以下の2方式について整理した。

### Lightsail Instance方式

AWS上にUbuntuなどのLinuxサーバーを作成し、その中へ自分でDockerを導入して、OCRアプリを動かす方式である。

```text
Instance作成
↓
Linuxサーバー起動
↓
SSH接続
↓
Docker導入
↓
Docker image配置
↓
OCRアプリ起動
```

Instance方式では、Linuxサーバーの内部を自由に操作できる。一方で、Linux、Docker、port、containerの再起動、ログなども自分で管理する必要がある。

### Lightsail Container Service方式

Dockerアプリを動かす基盤をAWS側に任せ、Docker imageとcontainerの実行設定を中心に管理する方式である。

```text
Docker imageを用意
↓
Lightsail Container Serviceへ配置
↓
container起動
↓
外部公開
```

Instance方式と比較すると、LinuxサーバーやDocker実行環境を自分で管理する範囲が小さく、Dockerアプリの実行に集中しやすい。

7月9日時点では、今回のOCRデモにはContainer Serviceが適している可能性が高いと考えた。ただし、この時点ではまだコード上で比較していなかったため、7月10日にAWS CDKを使って両方式を定義し、比較することとした。

## 2. 現時点の選定結果

今回の営業用OCRデモでは、現時点でLightsail Container Serviceを採用する方針としている。

今回のOCRアプリは、概ね以下の処理を行う小規模なWebアプリを想定している。

```text
画像を受け取る
↓
OCR処理を行う
↓
OpenAI APIを呼び出す
↓
結果を返す
```

現時点では、以下の特徴がある。

- Docker化を予定している
- DBへ大量のデータを保存し続ける予定がない
- サーバー内へ画像を長期間保存する予定がない
- OpenAI API呼び出しが処理の中心である
- 外部からアクセス可能なWeb APIとして公開する
- 特殊なLinux設定を行う予定がない

Instance方式でも実現は可能である。ただし、Instance方式ではOCRアプリに加えて、Ubuntu、Docker、containerの起動、port、ログなども管理する必要がある。

今回の目的はLinuxサーバーを細かく管理することではなく、Docker化したOCRアプリを小規模に公開することである。そのため、Container Serviceの方が今回の用途に適していると判断した。

## 3. 7月10日に行ったCDK比較実験

7月10日は、Container Serviceが適しているという判断をコード上でも確認するため、`lib/ocr-app-stack.ts`にInstance方式とContainer Service方式の両方を定義した。

確認したCDKの構造は以下である。

```text
CDK App
↓
OcrAppStack
↓
CfnInstance / CfnContainer
↓
cdk synth
↓
AWS向けの構成定義（CloudFormation形式）
```

ここで使用している`CfnInstance`と`CfnContainer`は、コードを書いた時点でAWS上に本物のリソースを作成するものではない。CDKコード上に、AWSへ作成する構成の設計情報を登録するものである。

その後、以下のコマンドを実行した。

```bash
cdk synth
```

`cdk synth`は、TypeScriptで書いたCDKコードを、AWSが読み取れるCloudFormation形式の構成定義へ変換するコマンドである。

**現時点では`cdk deploy`は行っていないため、AWS上へ実際のInstanceやContainer Serviceを作成した段階ではない。**

`cdk synth`の成功は「AWS向け構成へ変換できた」ことであり、「AWSへ配置できた」ことではない。

## 4. lib/ocr-app-stack.tsの実装内容

### 4.1 Lightsail Instance方式

`lib/ocr-app-stack.ts`では、比較対象として`CfnInstance`を定義している。実装の主要部分は以下である。

```typescript
new lightsail.CfnInstance(this, 'OcrDemoInstance', {
  instanceName: 'ocr-demo-instance',
  availabilityZone: 'ap-northeast-1a',
  blueprintId: 'ubuntu_22_04',
  bundleId: 'nano_3_0',
  networking: {
    ports: [
      { fromPort: 22, toPort: 22, protocol: 'tcp' },
      { fromPort: 80, toPort: 80, protocol: 'tcp' },
    ],
  },
});
```

| 設定項目 | 内容 |
|---|---|
| `instanceName` | AWS上で作成するInstanceの名前 |
| `availabilityZone` | Instanceを配置する場所（東京リージョン内） |
| `blueprintId` | 使用するOS（Ubuntu 22.04） |
| `bundleId` | Instanceの大きさ（nano） |
| port 22 | SSH接続用 |
| port 80 | HTTP公開用 |

このコードで定義している中心は、OCRアプリそのものではなく、OCRアプリを動かすためのLinuxサーバーである。Instance作成後は、SSH接続、Docker導入、image配置、container起動を別途行う必要がある。

### 4.2 Lightsail Container Service方式

`lib/ocr-app-stack.ts`では、Container Service方式として`CfnContainer`を定義している。

現在の実装では、Docker imageやportを`CfnParameter`（CloudFormation実行時に外部から値を渡せる仕組み）で定義している。

```typescript
const containerImage = new cdk.CfnParameter(this, 'ContainerImage', {
  type: 'String',
  default: 'nginx:latest',
  description: 'Lightsail Container Serviceで実行するDocker Image',
});

const containerPort = new cdk.CfnParameter(this, 'ContainerPort', {
  type: 'Number',
  default: 80,
  description: 'Container内部でアプリが待ち受けるport',
});
```

Container Serviceの定義は以下である。

```typescript
new lightsail.CfnContainer(this, 'OcrDemoContainerService', {
  serviceName: 'ocr-demo-container-service',
  power: 'nano',
  scale: 1,
  containerServiceDeployment: {
    containers: [
      {
        containerName: 'ocr-api',
        image: containerImage.valueAsString,
        environment: [
          { variable: 'OPENAI_VISION_MODEL', value: 'gpt-4.1-mini' },
        ],
        ports: [
          { port: containerPort.valueAsString, protocol: 'HTTP' },
        ],
      },
    ],
    publicEndpoint: {
      containerName: 'ocr-api',
      containerPort: containerPort.valueAsNumber,
      healthCheckConfig: {
        path: '/',
        intervalSeconds: 10,
        timeoutSeconds: 2,
        healthyThreshold: 2,
        unhealthyThreshold: 2,
        successCodes: '200-399',
      },
    },
  },
});
```

| 設定項目 | 内容 |
|---|---|
| `serviceName` | Container Serviceの名前 |
| `power` | containerを動かすための処理能力（nano） |
| `scale` | 同時に動かす実行数（1） |
| `containerName` | containerの名前（`ocr-api`） |
| `image` | 起動するDocker image（現在は`nginx:latest`） |
| `environment` | containerへ渡す環境変数（`OPENAI_VISION_MODEL=gpt-4.1-mini`） |
| `ports` | containerが使用するport（現在は80） |
| `publicEndpoint` | 外部へ公開するcontainerとport |
| `healthCheckConfig` | containerが正常に動いているか確認する設定 |

また、`CfnOutput`として、deploy後にContainer Service名と公開URLを出力する設定が含まれている。

**備考：** `healthCheckPath`パラメータが定義されているが、Container Serviceの`healthCheckConfig.path`に接続されていない（未使用状態）。これは今後OCR image移行時に修正が必要な箇所である。

## 6. 比較して分かったこと

| 比較項目 | Instance方式 | Container Service方式 |
|---|---|---|
| 中心となるもの | Linuxサーバー | Dockerアプリ |
| Docker導入 | 自分で行う | AWS側の基盤を利用 |
| アプリの指定 | Instance作成後に配置 | container設定内で指定 |
| 外部公開 | 自分で管理する範囲が広い | Public Endpointを設定 |
| 正常動作確認 | 自分で用意 | Health Checkを設定できる |
| 管理作業 | 多い | 比較的少ない |
| 今回への適合 | 実現可能 | より適している |
